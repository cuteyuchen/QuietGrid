# 04. 自适应网格算法

## 1. 设计目标

网格数量不应该是主要配置项。系统应先确定：可交易区间、最小有效间距、风险预算和最大库存，然后反推出网格数量与每格数量。

## 2. 输入与输出

### 输入

- 当前价格与中枢价格；
- ATR、实现波动率和趋势强度；
- spread、资金费、Maker/Taker 费率和滑点估计；
- Regime Score；
- 账户权益与会话风险预算；
- 最大库存、最大名义仓位和最小订单规则；
- 距强制离场时间。

### 输出

```python
@dataclass(frozen=True)
class GridPlan:
    center: Decimal
    lower: Decimal
    upper: Decimal
    step_pct: Decimal
    grid_prices: tuple[Decimal, ...]
    qty_by_level: tuple[Decimal, ...]
    max_inventory_notional: Decimal
    hard_stop_price: Decimal
    expires_at: datetime
    parameter_version: str
```

## 3. 中枢与区间

建议使用稳健中枢，而不是仅使用观察期简单均值：

```text
center = EWMA(close, half_life) 或 rolling_median
```

半区间宽度可以取多个估计中的较大值：

```text
atr_band      = k_atr * ATR
quantile_band = max(center - q_low, q_high - center)
sigma_band    = k_sigma * center * sigma
half_width    = max(atr_band, quantile_band, sigma_band, min_range_abs)
```

然后应用时间与风险缩放：

```text
half_width *= regime_range_multiplier
half_width = min(half_width, max_range_pct * center)
```

如果区间过宽、当前价已远离中枢、或剩余交易时间不足以覆盖合理成交周期，则放弃建网格。

## 4. 成本地板

每格毛价差必须大于完整成本和安全垫：

```text
cost_floor_pct =
    maker_fee_open
  + maker_fee_close
  + expected_funding_cost
  + expected_adverse_selection
  + price_rounding_cost
  + safety_margin
```

即使当前 Maker 费为 0，也要运行时验证，并为费率变化留出安全垫。

## 5. 动态网格间距

建议：

```text
step_pct = max(
    cost_floor_pct,
    k_atr_step * ATR_pct,
    k_sigma_step * sigma_short
)
```

然后限制：

```text
step_pct = clamp(step_pct, min_step_pct, max_step_pct)
```

ATR 和波动率扩大时，网格变宽；市场平静时可以变密，但不能低于成本地板。

## 6. 网格数量计算

对于等比网格：

```text
N_raw = floor( ln(upper / lower) / ln(1 + step_pct) )
N = clamp(N_raw, min_grid_num, max_grid_num)
```

重新计算实际间距，保证上下沿和格数一致：

```text
actual_step = exp(ln(upper/lower)/N) - 1
```

若实际间距低于成本地板，则减少格数，而不是强行使用密集网格。

### 密集还是稀疏

不存在“越密越好”或“越少越好”。应比较：

```text
Expected Net PnL ≈ 成交次数 × 单格净利润 - 趋势与库存损失
```

密集网格的主要问题：

- 单格利润更薄；
- 对盘口噪声和逆向选择更敏感；
- 订单与对账压力更大；
- 更容易快速累积单向库存；
- 回测容易高估 Maker 成交。

稀疏网格的问题：成交少，资金利用率低。v2.0 应通过历史数据搜索稳健区间，而不是追求最大成交次数。

## 7. 非等距网格

距离中枢越远，可逐渐扩大间距，减少趋势行情中的连续补仓：

```text
step_i = base_step * (1 + expansion_rate * |i|)
```

或：

```text
price_i = center * exp(sign(i) * base_log_step * |i|^p), p > 1
```

初期建议先实现线性扩张，便于解释和回测。

## 8. 每格仓位计算

先定义会话最大可承受损失：

```text
session_loss_budget = account_equity * session_risk_pct
```

对每一层估计从成交价到硬止损的损失距离：

```text
loss_per_unit_i = abs(entry_price_i - hard_stop_price)
```

风险约束数量：

```text
qty_by_risk_i = allocated_risk_i / loss_per_unit_i
```

名义约束数量：

```text
qty_by_notional_i = level_notional_cap / entry_price_i
```

实际数量：

```text
qty_i = min(qty_by_risk_i, qty_by_notional_i, inventory_remaining_i)
```

每格等额名义仓位虽然简单，但无法保证远端网格在止损时风险可控。v2.0 推荐风险等权或靠近中枢稍大、远端递减。

## 9. 动态重算

滚动重算不等于每分钟撤掉全部订单。建议：

- 每分钟更新特征和风险；
- 每 15–30 分钟评估是否需要重建；
- 只有当中枢漂移、间距变化或 Regime Score 超过阈值时才重建；
- 重建必须先冻结新挂单、对账、撤销旧单，再原子性切换；
- 已成交库存不得因重建而丢失配对关系。

## 10. 伪代码

def build_grid_plan(snapshot, risk_budget, rules, config):
    assert snapshot.regime.allowed

    center = robust_center(snapshot.prices)
    half_width = calculate_half_width(snapshot, config)
    lower, upper = center - half_width, center + half_width

    cost_floor = estimate_total_cost(snapshot, rules, config)
    step = max(
        cost_floor,
        config.k_atr_step * snapshot.atr_pct,
        config.k_sigma_step * snapshot.sigma_short,
    )
    step = clamp(step, config.min_step_pct, config.max_step_pct)

    n = int(math.log(upper / lower) / math.log(1 + step))
    n = clamp(n, config.min_grid_num, config.max_grid_num)
    actual_step = math.exp(math.log(upper / lower) / n) - 1

    if actual_step < cost_floor:
        raise GridRejected("net edge below cost floor")

    prices = generate_expanding_grid(center, lower, upper, n, config)
    quantities = allocate_by_risk(prices, risk_budget, snapshot, rules, config)
    validate_exchange_constraints(prices, quantities, rules)

    return GridPlan(...)
