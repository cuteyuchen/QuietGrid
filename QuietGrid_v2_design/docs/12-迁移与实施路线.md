# 12. 迁移与实施路线

## 1. 原则

不进行“大爆炸重写”。现有系统已完成交易所、状态机、Web 和测试网链路，应采用绞杀者模式逐模块替换。

## 2. Phase 0：冻结基线

- 为当前代码打 tag；
- 保存当前测试结果和配置；
- 将原始计划与 README 归档；
- 明确当前实盘尚未完成的验收；
- 禁止在文档重构期间提高真实风险。

## 3. Phase 1：事件与版本基础

新增：

- 统一事件定义；
- `event_store`；
- config version；
- code commit 记录；
- 数据时间字段。

不改变现有交易策略，先验证记录与重放。

## 4. Phase 2：Regime Engine

- 把现有波动率、ATR、趋势和流动性计算封装为 Feature Engine；
- 实现透明规则评分；
- 初期只做“旁路影子模式”，记录建议但不控制交易；
- 对比其过滤效果；
- 验证后让 Regime Engine 获得启动门控权。

## 5. Phase 3：Inventory Manager

- 引入 inventory lots；
- 对当前网格成交配对逻辑进行审计；
- 先在 UI 展示库存利用率；
- 再实现同向挂单抑制；
- 最后实现主动降仓。

## 6. Phase 4：统一 Risk Manager

将散落在 controller、grid_engine 和 risk.py 中的判断收敛为统一接口：

```python
risk_decision = risk_manager.evaluate(context, intent)
```

动作：`ALLOW / REDUCE / BLOCK / CLOSE / HALT_WINDOW`。

实现窗口损失、连续止损和冷却恢复门控。

## 7. Phase 5：Adaptive Grid Generator

- 保留当前等比网格作为 `legacy`；
- 新增 `adaptive_v2`；
- 同一历史数据并行回测；
- 不直接替换实盘；
- 通过样本外后在 Mock/测试环境影子运行。

## 8. Phase 6：回测升级

- 未来数据断言；
- 保守 Maker 成交模型；
- Walk-Forward；
- Monte Carlo；
- 报告版本化；
- Strategy Replay。

## 9. Phase 7：前端改造

优先级：

1. Dashboard 安全状态；
2. Regime 与 Inventory；
3. Risk Center；
4. 网格可视化；
5. 回测与 Replay；
6. 设置版本与运维。

## 10. Phase 8：验收

依次完成：Mock → 回测 → 样本外 → paper → 测试网链路 → 极小资金 1 倍单标的 → 多窗口观察。

## 11. 退出与回滚

每个新模块有 feature flag：

```yaml
features:
  regime_v2: false
  inventory_manager: false
  adaptive_grid_v2: false
  risk_manager_v2: false
```

出现问题时可回滚到 legacy，但风险降低型熔断和交易所端止损不可被 feature flag 关闭。
