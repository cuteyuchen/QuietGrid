# QuietGrid BTC/ETH 两年历史数据与稳健性回测

本文档描述 `scripts/robust_backtest.py` 的研究协议。该工具与 Trader 完全隔离：
不读取 API Key、不连接测试网账户、不下单，只下载公开归档并调用现有网格求解器和
逐 Bar 回测器。

## 1. 目标和非目标

目标：

- 下载 Binance 官方 USDⓈ-M `BTCUSDT`、`ETHUSDT` 1 分钟 K 线；
- 优先使用月归档，月包缺失时回退日归档；
- 每个 ZIP 必须通过官方 `.CHECKSUM`，再冻结为不可变 CSV；
- 只研究 NYSE 周末或长假休市窗口，排除普通工作日隔夜；
- 严格划分开发集、验证集、最终 OOS；
- 用 Walk-Forward、参数邻域、Maker 成交概率压力测试判断稳健性；
- 同时输出机器可读 JSON 和人工可读 Markdown。

非目标：

- 不保证未来收益；
- 不用最终 OOS 反复调参；
- 1 分钟 OHLC 无法模拟真实 Maker 排队位置；
- 当前 Binance K 线归档不含本工具需要的资金费 sidecar，因此使用显式的固定资金费
  压力假设，报告会标明这一限制。

## 2. 下载并冻结近两年数据

在仓库根目录执行：

```powershell
.\.venv\Scripts\python.exe -m scripts.robust_backtest download `
  --symbols BTCUSDT,ETHUSDT `
  --start 2024-07-19T00:00:00Z `
  --end 2026-07-19T00:00:00Z `
  --output-dir data/backtests/robustness
```

如果网络需要代理：

```powershell
.\.venv\Scripts\python.exe -m scripts.robust_backtest download `
  --symbols BTCUSDT,ETHUSDT `
  --start 2024-07-19T00:00:00Z `
  --end 2026-07-19T00:00:00Z `
  --proxy-url http://127.0.0.1:7897
```

归档通常有一天生成延迟。工具只冻结已经进入官方日归档的完整日期，并在 manifest
中设置：

```json
{
  "tail_truncated": true,
  "archive_available_end": "...",
  "official_checksums_verified": true
}
```

每个标的生成：

```text
data/backtests/robustness/
├── binance_um_btcusdt_1m_<起点>_<终点>_<hash>.csv
├── binance_um_btcusdt_1m_<起点>_<终点>_<hash>.manifest.json
├── binance_um_ethusdt_1m_<起点>_<终点>_<hash>.csv
└── binance_um_ethusdt_1m_<起点>_<终点>_<hash>.manifest.json
```

冻结阶段验证：

- 时间严格递增；
- 重复行内容必须完全相同；
- 冲突重复立即失败；
- 时间必须落在 1 分钟边界；
- 缺口比例默认不得超过 `0.1%`；
- 冻结 CSV 计算 SHA-256；
- manifest 保存每个月包/日包的官方 checksum 和本地 checksum。

可单独复核：

```powershell
$manifests = Get-ChildItem data/backtests/robustness -Filter '*.manifest.json'
.\.venv\Scripts\python.exe -m scripts.robust_backtest verify @manifests
```

## 3. 运行回测

把两个实际 manifest 路径传入：

```powershell
.\.venv\Scripts\python.exe -m scripts.robust_backtest backtest `
  data/backtests/robustness/<btc-manifest>.manifest.json `
  data/backtests/robustness/<eth-manifest>.manifest.json `
  --capital 500 `
  --direction-modes NEUTRAL `
  --range-multipliers 0.75,1.0,1.25 `
  --min-steps 0.0012,0.0015,0.0018 `
  --stop-buffers 0.01,0.015,0.02 `
  --report-dir reports/robustness
```

如需分别研究三种方向，可显式传入：

```text
--direction-modes NEUTRAL,LONG,SHORT
```

这会扩大候选空间，不应在看过最终 OOS 后临时开启。

### 3.1 开发/验证诊断（不读取最终 OOS）

逐窗口损失拆解会分别报告配对网格盈亏、库存退出盈亏和退出费用。这里的
`paired_grid_pnl` 明确定义为：

```text
paired_grid_pnl = gross_grid_pnl - stop_exit_pnl
```

长周期动态方向诊断使用决策时已经闭合的 24h/72h 历史，将阶段分为：

- 持续趋势：选择同向（MOMENTUM）或反向（CONTRARIAN）方向网格；
- 低趋势比率：选择中性网格；
- 模糊阶段：暂停。

```powershell
$manifests = Get-ChildItem data/backtests/robustness -Filter '*.manifest.json'
.\.venv\Scripts\python.exe -m scripts.robust_backtest diagnose-dynamic-mode `
  @manifests `
  --lookbacks 1440,4320 `
  --trend-alignments MOMENTUM,CONTRARIAN `
  --wind-down-bars 1440 `
  --max-inventory-notional 200 `
  --report-dir reports/robustness
```

该命令同时输出 LONG、SHORT、NEUTRAL 固定模式基线。所有候选只在开发集和
验证集上计算，JSON 中最终 OOS 必须保持 `SEALED_NOT_EVALUATED`。

## 4. 严格研究协议

### 4.1 周末窗口

只有相邻两个 NYSE 交易日的日期间隔大于一天时才创建窗口：

```text
前一交易日收盘
→ 观察 180 根已闭合 1m K 线
→ 网格运行
→ 下一交易日 04:00（纽约时间）前 120 分钟强制离场
```

因此，周一收盘到周二盘前之类的普通隔夜不会进入回测。节假日长周末会自然纳入。
窗口内任何一分钟缺失都会跳过该窗口，避免把不连续行情当成稳定低波动。

### 4.2 时间切分

所有标的共用相同的窗口 ID 边界，按时间严格切为：

- 开发集：前 50%；
- 验证集：随后 25%；
- 最终 OOS：最后 25%。

开发集用于全参数排序。只有开发集前 `K=5` 候选进入最终锁定过程；锁定分数取：

```text
min(开发集目标值, 验证集目标值, 验证集参数邻域中位数)
```

这会优先选择对轻微参数变化不敏感的“平台”，而不是开发集单点最高峰。

最终 OOS 在参数锁定后才运行。之后的邻域和成交概率运行是压力诊断，不得回流选参。
如果最终 OOS 失败，应修改策略假设、滚动到新的未来保留集并创建新研究版本；禁止在
同一 OOS 上“调到通过为止”。

### 4.3 无未来数据

- 入场只使用窗口开始后的前 180 根已闭合 K 线；
- 交易 Bar `t` 的留存 Regime 分只使用 `t` 之前的 180 根 K 线；
- 最终 OOS 不参与参数锁定。
- 动态方向只使用市场关闭前历史和观察期内已闭合 K 线；窗口之后的价格路径不参与
  模式选择。

### 4.4 执行与成本假设

默认：

```text
本金/标的                 500 USDT
杠杆                      1x
Maker 单边费率             0.02%
Taker 单边费率             0.05%
种子/止损滑点              10 bps
Maker 成交概率             65%
单 Bar 最多成交            2 笔
资金费压力假设             0.01% / 8h
```

网格参数仍由生产 `AdaptiveGridGenerator` 生成，撮合由生产
`run_grid_backtest()` 执行。历史盘口不可得，因此点差和深度为报告中的显式假设；
不能把 L0 回测结果当作真实排队成交证明。

## 5. 稳健性门槛

默认全部满足才标记为 `passed`：

- 最终 OOS 总盈亏为正；
- 最终 OOS Profit Factor ≥ 1.05；
- 最终 OOS 最大回撤 ≤ 5%；
- 实际交易覆盖率 ≥ 25%；
- 单一最佳窗口不超过全部正收益的 35%；
- 最终 OOS 参数邻域至少一半为正；
- Walk-Forward 拼接结果为正。

“通过”只说明在当前冻结数据和显式成本假设下未发现明显单点过拟合，不是收益保证。

## 6. 输出

JSON 包含：

- 数据集 ID、SHA-256、时间范围、官方 checksum 状态；
- 开发/验证/OOS 的精确窗口边界；
- 锁定参数；
- 全参数开发和验证指标；
- 最终 OOS 每个标的/窗口结果及阻断原因；
- Walk-Forward 每折选择和测试指标；
- 参数邻域与 50%/65%/80% Maker 成交概率压力测试；
- 每一项稳定性门槛的布尔结果。

Markdown 是同一 JSON 的摘要，不应作为唯一审计材料。

## 8. 2026-07-19 研究结论

在修复数量权重、双向止损、终场停止开仓和库存约束后：

- 固定 LONG：开发 `-66.21 USDT`，验证 `-25.90 USDT`；
- 固定 SHORT：开发 `-58.12 USDT`，验证 `-25.63 USDT`；
- 固定 NEUTRAL：开发 `-1.35 USDT`，验证 `+13.80 USDT`；
- 24h/72h 趋势延续与反向动态方向规则共 48 组，`0/48` 同时通过开发和验证门槛。

因此当前证据不支持方向种子网格具有稳健历史优势，也不支持继续围绕长周期方向阈值
调参。最终 OOS 未执行。后续策略研究必须提出新的库存收益机制，并使用新的未来数据或
影子运行作为真正独立验证，不能把同一保留集反复调到通过。

## 9. 2026-07-20 最终 OOS 与后续假设

锁定的 BTC/ETH 联合策略在 `2026-01-02` 至 `2026-07-17` 最终 OOS 上一次性评估：

## 10. 2026-07-21 标的独立库存策略联合研究

在不读取已消费最终 OOS 的前提下，使用同一份两年冻结数据重新进行 BTC/ETH 联合
10-seed 研究。开发集 56 个窗口，验证集 28 个窗口；最终 OOS 29 个窗口在本轮报告中
保持 `SEALED_NOT_EVALUATED`，表示本轮没有重新读取。由于这些窗口此前已被旧策略一次性
消费，它们不能再作为本策略的独立 OOS。

本轮锁定的标的独立策略为：

| 标的 | 参数 | 入口过滤器 | 本金 | 最大库存 | 单侧未配对上限 | 减仓目标 |
|---|---|---|---:|---:|---:|---:|
| BTCUSDT | `r1.25 / step 0.0018` | 无 | 500 | 200 | 1 层 | 0.50 格 |
| ETHUSDT | `r1.00 / step 0.0018` | `de0.50_ve1.05_rr0.25` | 300 | 120 | 0 层 | 1.00 格 |

其中减仓目标表示 OPEN 成交后，REDUCE 订单使用完整网格步长的固定比例；`1.00` 保持
原行为。回测缓存键同时包含这两个标的独立参数，避免不同策略复用错误结果。

严格联合门槛除组合指标外，还要求每个 seed、每档成本、开发/验证拆分中的 BTC 和 ETH
分别满足正收益、Profit Factor ≥ 1.05、最大回撤 ≤ 5% 和交易覆盖率 ≥ 25%。结果如下：

| 费用场景 | 通过 Seed | 通过率 | 开发盈亏范围 | 验证盈亏范围 |

### 10.1 生产一致性缺口

截至本报告生成时，该候选尚未与真实交易路径完全对齐：

- `GridEngine.handle_order_filled()` 仍把 OPEN 成交后的 REDUCE 单放在下一完整网格价；
- 生产 `grid` 配置仍是全局 `k_atr_range/k_sigma_range=2.0`、`min_step_pct=0.0015`，
  尚不能分别表达 BTC `r1.25` 与 ETH `r1.00`；
- ETH 的 `de0.50_ve1.05_rr0.25` 入口过滤器尚未接入生产 Regime 准入路径。

因此不得把本轮回测通过等同于当前 Trader 已采用该策略。下一开发阶段必须先实现按标的
策略配置，并用同一组冻结窗口证明生产引擎与研究回测的订单价格、准入结果和盈亏一致；
完成一致性验证后，才可进入不下真实资金订单的测试网影子运行。

### 10.2 撤单延迟保守复验

原 `INTRABAR` 库存上限会在同一根 1 分钟 Bar 的第一层 OPEN 成交后，立即阻止第二层
同侧 OPEN 成交；真实交易所无法保证控制器能在同一分钟内先于后续成交完成撤单。为消除
这一乐观假设，新增 `BAR_BOUNDARY` 模式：同一 Bar 开始时尚未达到上限，则该 Bar 内已经
触及的多层 OPEN 均允许成交，库存上限从下一 Bar 起生效。

使用完全相同的 BTC/ETH 参数、10 个 seed 和三档费用重跑后：

- BASE、COST25、COST50 通过率仍为 `100% / 100% / 80%`；
- 最差组合拆分盈亏仍为 `+5.69 USDT`；
- BTC/ETH 的 COST50 最差拆分仍为 `+1.61 / +1.14 USDT`；
- seed 59、127 的开发集确实产生不同成交路径，组合盈亏约增加 `0.20～0.21 USDT`，
  证明该压力模型不是空操作；其余拆分保持一致。

摘要见 `reports/robustness/btc-eth-joint-seeds-bar-boundary-20260721.md`。这一结果消除了
“同 Bar 即时撤单”这一项已知乐观偏差，但仍只属于重复使用开发/验证数据的稳健性证据，
不会恢复已经消费的最终 OOS 独立性。
